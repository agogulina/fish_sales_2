# Промо-копайлот — модульный Python-проект

Причинная аналитика и оптимизация промо-скидок для производителя рыбной продукции.
Код разбит **по смыслу** на модули (`src/`) и точки входа (`scripts/`); каждый слой запускается отдельным скриптом в фиксированном порядке.

```
поиск товаров (Слой 1) → причинный эффект скидки (Слой 2) → оптимизатор промо (Слой 3)
```

---

## Структура проекта

```
promo-copilot-py/
├── config.py                  # имена столбцов, пути, модели, гиперпараметры
├── requirements.txt
├── .gitignore
│
├── src/                       #   логика
│   ├── data.py                #   загрузка, помесячная панель, каталог
│   ├── features.py            #   лаги/сезонность, парсер веса, продукты, эмбеддинги e5
│   ├── layer1_search.py       #   Слой 1: 6 методов поиска, метрики, значимость
│   ├── cannibalization.py     #   проверка каннибализации (within, FE, Double ML)
│   ├── layer2_causal.py       #   Слой 2: DML, CATE, доза-отклик, конформные интервалы, плацебо
│   ├── pricing.py             #   сопоставление цен (вес → руб/кг → руб/шт)
│   └── layer3_optimizer.py    #   Слой 3: экономика, оптимизатор (PuLP), off-policy
│
├── scripts/                   
│   ├── run_pricing.py
│   ├── run_layer1.py
│   ├── run_cannibalization.py
│   ├── run_layer2.py
│   └── run_layer3.py
│
├── data/                      
│   └── README.md
└── outputs/                   # артефакты 
```

### За что отвечает каждый модуль

| Модуль | Смысл |
|--------|-------|
| `config.py` | единый источник констант: столбцы, пути, названия моделей, сетки скидок, гиперпараметры |
| `src/data.py` | чтение CSV, агрегация в помесячную панель `SKU×клиент×месяц`, каталог товаров |
| `src/features.py` | временные признаки (лаги/скользящие/сезонность), парсер веса из наименования, словарь продуктов, эмбеддинги e5 (опционально) |
| `src/layer1_search.py` | поиск: лексический, категорийный baseline, плотный e5, e5+реранкер, структурный; метрики MAP/R@10/nDCG; значимость (Фридман/Неменьи) |
| `src/cannibalization.py` | граф субститутов и три оценивателя влияния скидки соседа |
| `src/layer2_causal.py` | причинная оценка эффекта скидки и всё вокруг неё |
| `src/pricing.py` | привязка реальных цен к нашим SKU |
| `src/layer3_optimizer.py` | экономика вклада в прибыль и подбор скидок под ограничения |

## Установка

```bash
pip install -r requirements.txt
```

Тяжёлые библиотеки (`sentence-transformers`, `faiss`, `econml`, `dowhy`, `mapie`, `pulp`, `statsmodels`, `pymorphy3`, `rapidfuzz`) **опциональны**: при их отсутствии модули используют встроенные запасные реализации (например, DML по кластерному бутстрэпу вместо точных SE, жадный оптимизатор вместо PuLP, поиск без e5). Для полной воспроизводимости результатов нужен полный список.

## Данные

Положите в `data/` три файла (формат — в `data/README.md`):
`sales.csv`, `promo_sales1.csv`, `promo_final.csv`. Для Слоя 3 также `lenta_prices.xlsx` (прайс сети). Данные в репозиторий не коммитятся (см. `.gitignore`).

## Запуск всего конвейера одной командой

В корне есть `__main__.py` — диспетчер всех шагов:

```bash
python __main__.py all              # все слои по порядку: pricing -> layer1 -> cannibalization -> layer2 -> layer3
python __main__.py layer2           # только один слой
python __main__.py pricing          # только ценовая таблица
```

Шаг без входных данных аккуратно пропускается с сообщением, а не роняет весь конвейер.

## Как запускать — порядок

Запускать из **корня проекта**. Порядок важен: цены и поиск — независимы, причинность — перед оптимизатором.

```bash
# Шаг 0 — ценовая таблица (нужна Слою 3). Требует data/lenta_prices.xlsx
python scripts/run_pricing.py
#   -> outputs/prices_sku_matching.xlsx

# Шаг 1 — Слой 1: семантический поиск (метрики + значимость)
python scripts/run_layer1.py

# Шаг 1b — проверка каннибализации (исследовательский блок)
python scripts/run_cannibalization.py

# Шаг 2 — Слой 2: причинный отклик на скидку
python scripts/run_layer2.py

# Шаг 3 — Слой 3: оптимизатор промо (нужны Шаг 0 и логика Слоя 2)
python scripts/run_layer3.py
#   -> outputs/promo_plan_layer3.xlsx
```

**Зависимости по данным между шагами:**
- `run_layer3.py` использует `outputs/prices_sku_matching.xlsx` из `run_pricing.py` → запускайте Шаг 0 раньше Шага 3.
- `run_layer2.py` и `run_layer3.py` сами считают причинный эффект (общий модуль `layer2_causal`), поэтому Слой 2 не обязан запускаться перед Слоем 3, но логически идёт раньше.
- `run_layer1.py` и `run_cannibalization.py` независимы от цен.

Минимальный сценарий «всё по порядку»:
```bash
python scripts/run_pricing.py && \
python scripts/run_layer1.py && \
python scripts/run_cannibalization.py && \
python scripts/run_layer2.py && \
python scripts/run_layer3.py
```

---

## Что выводит каждый скрипт

- **run_pricing** — сводка покрытия и файл цен по SKU (вес, руб/кг, наценка).
- **run_layer1** — таблица метрик по методам, тест Фридмана, средние ранги, демо `find_products`.
- **run_cannibalization** — оценки эффекта субститутов тремя методами + позитивный контроль.
- **run_layer2** — `theta` эффекта скидки с CI, CATE по категориям, кривая доза-отклик, обратная задача, конформное покрытие, плацебо.
- **run_layer3** — причинный эффект по категориям, план промо, сравнение с историей при равном бюджете, файл `promo_plan_layer3.xlsx`.

## Ключевые результаты (на данных исследования)

- **Слой 1:** лучший метод e5 + реранкер, MAP ≈ 0.49 (вдвое выше baseline), различия значимы (Фридман p = 0.0014).
- **Каннибализация:** не обнаружена (положительный знак во всех трёх оценивателях) → ограничение для Слоя 3.
- **Слой 2:** +10 п.п. скидки → ≈ +33% продаж (θ ≈ +0.029, p ≈ 0); доза-отклик с убывающей отдачей; плацебо ≈ 0; конформное покрытие 91.4%.
- **Слой 3:** при равном бюджете скидок план даёт ≈ +143% вклада в прибыль против исторической политики.

## Ограничения

Данные помесячные; себестоимость в Слое 3 — сценарный параметр `COST_SHARE` (нет реальной); цены — по одной сети, по продукту; каннибализация требует недельных данных.
